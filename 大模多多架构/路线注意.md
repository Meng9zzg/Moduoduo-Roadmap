## 一、必须关注的行业协议

### 1.1 A2A 协议（Agent-to-Agent）

| 指标 | 内容 |
|------|------|
| **制定方** | Google 主导，已捐给 Linux Foundation（行业公共标准） |
| **发布时间** | 2025.04.09 |
| **当前版本** | v0.3（2025.07.30） |
| **支持组织** | 150+（Salesforce / SAP / Deloitte 等） |
| **协议许可** | Apache 2.0 开源 |
| **实际落地** | Tyson Foods、Gordon Food Service 供应链协作 |

**解决什么问题：Agent 与 Agent 之间的互操作**

不同框架、不同厂商构建的 AI Agent 之间，如何互相发现、通信、协作。

```
没有 A2A：                         有 A2A：

Agent A ─?─ Agent B              Agent A ←──→ Agent B
   （需要人工写集成代码）            （标准协议自动通信）
```

**类比理解：**
- OpenAI API 格式 = 所有餐厅用同一种点菜单格式（应用调模型的标准）
- A2A 协议 = 不同餐厅之间能互相转单（Agent 间协作的标准）

**对大模多多的影响：**
- 短期（2026 上半年）：不需要立即接入，QAgent 是单 Agent 交互系统
- 中期（2026 下半年-2027）：A2A 如果到 v1.0 且被广泛采纳，QAgent 需要评估是否支持
- 长期风险：A2A 可能像 OpenAI API 格式一样变成"不支持就出局"的行业标准。当年不兼容 OpenAI API 格式的模型服务商都被边缘化了，未来不支持 A2A 的 Agent 平台可能面临同样局面

**行动项：**
- [ ] 关注 A2A v1.0 发布时间（预计 2026 下半年-2027）
- [ ] 评估 QAgent 是否需要对外暴露 A2A 接口（多站点部署 / 接入甲方已有 AI 系统时）
- [ ] 关注 MCP + A2A 的组合使用模式

### 1.2 MCP 协议（Model Context Protocol）

| 指标 | 内容 |
|------|------|
| **制定方** | Anthropic |
| **状态** | 已成行业标准 |
| **规模** | 10,000+ 服务器，9,700 万月下载 |
| **采纳方** | OpenAI / Google / Microsoft 统一采纳 |

**解决什么问题：Agent 调用外部工具的标准化**

```
A2A vs MCP 的关系：

MCP（纵向）                    A2A（横向）

  Agent                    Agent A ←──→ Agent B
    │                         ↕            ↕
    ▼                      Agent C ←──→ Agent D
 工具/API/数据库
```

- MCP = Agent 怎么调工具（纵向）
- A2A = Agent 怎么互相协作（横向）
- 两者互补，不冲突

**对大模多多的影响：**
- Agent Flow 和 Agent Team 均不支持 MCP——这是它们被判定"技术淘汰风险高"的重要原因之一
- QAgent 目前直接调模型 API，暂不需要 MCP
- 但如果未来 QAgent 需要接入外部工具（企业知识库 / CRM / ERP），MCP 是最合理的接入方式

**行动项：**
- [ ] QAgent 暂不接入，持续关注
- [ ] 如果甲方要求"接入我们的企业知识库"，优先用 MCP 方案

## 二、产品线战略决策备忘

### 2.1 三条产品线的最终判断

```
┌──────────────────────────────────────────────────────────────┐
│              大模多多 AI Agent 产品线决策                       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ★ QAgent → All-in（唯一核心产品线）                          │
│  ├─ 2 个交付项目（香港中银 + 汕尾凤山）                       │
│  ├─ 自研硬件（算力塔 + 传感器总成 + 一体机仓体）              │
│  ├─ 自研 CV 引擎（Pose+Hand+Face+Emotion 四引擎融合）        │
│  ├─ 5 条独有业务链路                                          │
│  └─ 时间语义引擎（全球几乎无竞品）                            │
│                                                              │
│  △ Agent Flow → 冻结 + 抽离零件                               │
│  ├─ 抽离 ModuoduoPro 网关为独立微服务                         │
│  ├─ 10 项软著独立归档，投标使用                               │
│  └─ 不再投入新功能开发                                        │
│                                                              │
│  ▽ Agent Team → 直接归档                                      │
│  ├─ GitHub Archive                                            │
│  ├─ 技术认知文档化保留                                        │
│  └─ BP 中不展示                                               │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 决策依据速查

| 产品线 | 上游 | 自研增量 | 上游风险 | 结论 |
|--------|------|---------|---------|------|
| **QAgent** | 无（全自研） | 100% | 无 | All-in |
| **Agent Flow** | Flowise（被 Workday 收购 2025.08） | 4600+ i18n + 网关 + 10 软著（~2-3 月工作量） | 上游可能闭源 | 冻结 |
| **Agent Team** | ROMA（Sentient AGI $85M 活跃迭代） | 430+ i18n + UI（~3-4 周工作量） | 上游远强于 fork | 归档 |

### 2.3 为什么不自建多智能体框架

投入产出比不成立：
- CrewAI：45K⭐ / $18M 融资 / Fortune 500 客户 / 450M agents/月
- ROMA：5K⭐ / $85M 融资 / Sentient AGI 全职团队
- MetaGPT：65K⭐ / 中国团队
- LangGraph：$12.5B 生态 / Uber/LinkedIn 使用

**如果 QAgent 未来需要多 Agent 协作，直接用以上成熟框架，不要自建。**

## 三、技术淘汰风险监控清单

### 3.1 QAgent 低风险项（核心资产安全）

| 资产 | 淘汰风险 | 理由 |
|------|---------|------|
| CV 引擎（四引擎融合） | 🟢 低 | MediaPipe 是 Google 长期维护底层库，工程化集成是落地经验壁垒 |
| 体感游戏 + 皮影戏 | 🟢 低 | 内容资产不随技术迭代淘汰 |
| 时间语义引擎 | 🟢 低 | 解决 LLM 根本性缺陷，与模型版本无关 |
| 防幻觉工程化 | 🟢 低 | 方法论+工程实践，不依赖特定模型 |
| 自研硬件 | 🟢 低 | 物理资产 = 物理壁垒 |

### 3.2 需要关注的风险项

| 风险 | 紧急度 | 触发条件 | 应对 |
|------|--------|---------|------|
| A2A 成为强制标准 | ⚠️ 中期 | v1.0 发布 + 甲方要求 | 评估 QAgent 接入 A2A |
| MCP 成为工具接入标准 | ⚠️ 中期 | 甲方要求接入企业知识库/CRM | QAgent 通过 MCP 接入 |
| 通义千问/豆包 API 大幅变更 | 🟡 低 | 模型厂商 API breaking change | LiteLLM 统一层吸收 |
| MediaPipe 重大版本变更 | 🟢 极低 | Google 重构 MediaPipe | CV 引擎适配层隔离 |
| ComfyUI 生态变化 | 🟡 低 | SD/Flux 生态洗牌 | AI 拍照模块适配 |

## 五、BP 叙事注意事项

### 5.1 可以说的

> "我们深入研究了 Flowise（Workday 数亿美元收购）和 ROMA（$85M 融资）两大开源 AI 平台，完成了中文化定制和技术评估，拿到了 10 项软著。这让我们深刻理解了企业级 AI 平台的完整架构。但我们清醒地认为，平台框架层不是我们应该自建的——我们的壁垒在 QAgent 的交付经验、自研硬件和 CV 引擎。"

### 5.2 不要说的

- ❌ "我们自研了 AI Agent 编排平台，有 330+ 节点 / 60+ RBAC 权限"
- ❌ "我们自研了递归层级化多智能体框架"
- ❌ "我们的 Agent 平台超越 Flowise/Dify"

### 5.3 为什么要诚实

- 技术尽调会查 GitHub，Agent Flow 的 Flowise 代码结构、Agent Team 的 `sentientresearchagent` 包名都会暴露 fork 关系
- 投资人更看重"判断力"——知道什么不做，比什么都做更有说服力
- QAgent 的真实壁垒（2 个交付 + 自研硬件 + CV 引擎）足够强，不需要用 fork 项目来充数
