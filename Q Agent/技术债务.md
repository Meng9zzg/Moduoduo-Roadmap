# Q Agent 技术债务与工程缺口

> 来源：GPT 对模多多现有资产的映射分析 + 初版构想验证 + GPT 远程运维四层体征框架
> 日期：2026-03-04
> 说明：以下三个缺口是"真实存在的工程缺口"，应纳入工程路线优先解决

---

## 一、缺 Eval & Safety 中枢（评测与安全中心）

### 1.1 通俗理解

**现在的问题：** 引擎"好不好"，说不清楚。

- 用户问了 100 个问题，AI 答对了几个？→ 答不出
- 体感游戏完成率多少？10 个人玩，几个人玩到结束？→ 不知道
- AI 有没有说错话？说了几次？→ 不确定
- 这个月比上个月表现好了还是差了？→ 没有数据

### 1.2 定义

Eval 中枢 = 一套系统，自动告诉你"引擎跑得好不好"。

- 每次用户交互，自动记录：问了什么、答了什么、用了多久、花了多少钱（模型调用费）
- 自动统计 5 个核心指标：
  - 端到端延迟（P50/P95）
  - 单位成本（每任务 ¥ / 每会话 ¥）
  - 任务成功率
  - 安全事故率（内容/隐私/越权/注入）
  - 可复现率（同输入同版本输出稳定性）
- 每周出一份报告：这周比上周好了还是差了，差在哪里

### 1.3 为什么重要

- 没有 Eval：引擎"靠感觉在进步"
- 有了 Eval：引擎"靠数据在进步"
- 投资人问"你怎么保证引擎质量"时，能拿出数据

### 1.4 最低可行第一步

给每次 LLM 调用加一个日志，记录：用户问题、AI 回答、响应时间、用了哪个模型。先有数据，再谈分析。

## 二、缺统一 Trace/Telemetry（全链路追踪与遥测）

### 2.1 通俗理解

**现在的问题：** 设备出问题时，要远程猜是哪里坏了。

- 日志分散在不同文件里
- 不确定是摄像头没检测到人、ASR 没听到、LLM 超时，还是 TTS 挂了
- 花 2 小时排查，才能定位问题

### 2.2 定义

**Trace = 一个用户从"走近设备"到"交互结束"的完整链路，全部串在一起记录。**

```
用户走近 → CV检测到人(50ms) → ASR开始监听(200ms) → 用户说话 
→ ASR转文字(800ms) → LLM生成回答(1200ms) → TTS合成语音(400ms) 
→ 播放回答 → 用户离开
```

每一步：耗时多少毫秒、成功/失败、失败原因——全部自动记录，串成一条链。

**Telemetry（遥测）= 设备的"体检报告"自动上报。**

- CPU/GPU 温度
- 内存占用
- 网络延迟
- 当日崩溃次数

### 2.3 为什么重要

- 设备出问题时，5 分钟定位原因，而不是 2 小时
- 没出问题时，也能提前发现"快要出问题了"（如 GPU 温度持续升高）
- 当设备数达到 10 台、50 台时，没有这个根本管不过来

### 2.4 最低可行第一步

给每次完整交互加一个唯一 ID（`traceId`），让 CV、ASR、LLM、TTS 的日志都带上这个 ID。出问题时，用一个 ID 就能找到整条链路。

### 2.5 远程运维：缺的不是远控，是「可观测 + 告警闭环」（GPT 补充）

**向日葵 = 你能"进去看"。**  
但自动告警需要的是：**系统自己把健康状况变成数据，然后对异常触发通知。**

可以把运维拆成 4 层（"生命体"的体征系统）：

#### A. 心跳与资产清单（最基础）

每台机固定上报：

- 设备 ID、地点、版本号（软件/模型/配置）
- 在线状态（最后心跳时间）
- 当前模式（导览/对话/游戏/播报/空闲）

#### B. 体征指标（像人一样有"生命体征"）

**硬件体征（户外很关键）：**

- GPU/CPU 温度、显存占用、功耗、风扇转速
- 机柜温度、门禁（算力塔柜门被打开）
- UPS/电源状态（掉电、欠压）
- 屏幕亮度状态（自动亮度是否工作）

**网络体征：**

- 上行/下行、丢包、RTT
- 到核心服务/模型网关的可用性（成功率、耗时 P95）

**传感器体征：**

- 摄像头是否有帧、帧率
- 深度相机是否有 depth
- 麦克风是否有输入能量（RMS）
- Pose/YOLO/Omni 推理是否持续产出

#### C. 业务体验指标（"智能"的 KPI）

- 互动转化：靠近的人里有多少进入互动
- 平均互动时长
- 语音识别成功率、误触发率
- 游戏完成率/放弃率
- 播报触发次数、扫码带走次数

#### D. 告警与处置（闭环）

**告警分级：**

- **P0：** 设备离线 / 过热 / 存储满 / 推理挂掉
- **P1：** 网络抖动导致对话失败率升高
- **P2：** 体验下降（识别率下降、误触发升高）

**自动缓解动作（不一定要重启整机）：**

- 重启某个子进程（ASR / Pose / Omni）
- 切换降级模式（离线播报 + 本地脚本）
- 降帧率 / 降分辨率保命

→ 本质是把运维从"靠人盯"升级到"靠系统盯"。ISO 27001 的思路也强调"日志/监控 + 异常行为检测"，即把系统行为记录并用于识别问题。

**后续行动：** 可输出《一体机运维指标字典 v0.1》——字段、阈值、告警文案，直接落地。

## 三、缺 Runtime 真正产品化

### 3.1 通俗理解

**现在的问题：** 引擎是"你脑子里的系统"，不是"别人能部署的产品"。

部署一台新设备需要：
- 你亲自配环境
- 你亲自调参数
- 你亲自测试每个模块
- 你亲自写配置文件
- 出了问题你亲自修

→ 引擎离开你就跑不起来。

### 3.2 定义

**Runtime 产品化 = 把你脑子里的东西变成一套任何工程师都能部署的标准化系统。**

| 现在（在你脑子里） | 产品化之后（在系统里） |
|------------------|---------------------|
| 你知道配置文件该怎么写 | 有配置模板 + 文档，别人照着填就行 |
| 你知道哪些模型参数该调 | 有默认参数集，覆盖 80% 场景 |
| 你知道部署的先后顺序 | 有部署脚本，一键完成 |
| 你知道出了什么问题该看哪里 | 有错误码 + 排查指南 |
| 你知道新场景该改哪些东西 | 有"场景包"标准，只改场景包 |

### 3.3 为什么重要

- 你一个人只能同时服务 2–3 个项目；产品化后，普通工程师也能部署
- 交付时间从 2 周变成 3 天
- 这是"第 5 个项目成本降到第 1 个的 50%"的前提
- 这是从"项目公司"变成"引擎公司"的分水岭

### 3.4 最低可行第一步

下次部署新设备时，把你做的每一步记录下来，写成一份《部署文档》。这份文档就是产品化的种子。

## 四、优先级建议

| 优先级 | 缺口 | 理由 |
|--------|------|------|
| P0 | Trace/Telemetry | 设备多了管不过来，最紧急 |
| P1 | Eval | 中期，用数据驱动改进 |
| P2 | Runtime 产品化 | 持续做，每次部署都往前推一步 |

---

## 五、一句话总结

| 缺什么 | 大白话 |
|--------|--------|
| Eval | 给引擎装一个"体检仪表盘" |
| Trace | 给每次交互装一个"行车记录仪" |
| Runtime 产品化 | 把你脑子里的系统变成别人也能用的产品 |
